# Runtime validation

Generated wrappers can be emitted with Arktype or Valibot schemas as the source
of truth. Params are validated on the way in, rows are validated on the way out,
and types are derived via the validator's native inference helper. One
definition per thing, no drift.

This is an opt-in mode. By default, `sqlfu generate` emits plain TypeScript
types with zero runtime validation.

## Why runtime validation at the wrapper boundary

The generator has always made SQL to TypeScript a compile-time guarantee.
Runtime validation closes the loop:

- **Bad params fail loudly at the callsite.** Mistyped booleans, missing string
  args, enum typos throw a readable error before the SQL driver sees them.
- **Schema drift surfaces at the adapter boundary.** A column removed without
  regenerating, a newly-non-null field, a new enum variant: these become
  exceptions at the boundary, not silently-wrong objects reaching a component.

The schemas are used by the generated wrapper itself, not just re-exported for
consumers. The exported schemas are also usable for forms, RPC wire validation,
fixtures, and other boundary checks.

## Turning it on

```ts
// sqlfu.config.ts
export default {
  db: './db/app.sqlite',
  migrations: './migrations',
  definitions: './definitions.sql',
  queries: './sql',
  generate: {
    validator: 'arktype',
  },
};
```

After toggling, re-run `sqlfu generate`. Install the validator library yourself
if it is not already a dependency.

## Picking a validator

Both supported validators implement [Standard Schema](https://standardschema.dev),
so sqlfu can validate through the same generated runtime path.

- **`'arktype'`**: the primary path. Arktype uses TypeScript-like syntax as
  schema: `type({slug: 'string', title: 'string | null'})`. Its object-shaped
  definitions fit the future `sqlfu_types` idea: serializable metadata that can
  drive both runtime validation and generated TypeScript types.
- **`'valibot'`**: smaller functional runtime, useful if your project already
  uses Valibot and wants generated wrappers to match.

Whatever you pick, the public shape of the generated wrapper is identical: one
callable with `.Params`, `.Result`, and `.sql` attached. You can switch between
the supported validators and regenerate without touching callsites.

## What the generated file looks like

For a query `sql/find-post-by-slug.sql`:

```sql
select id, slug, title, status from posts where slug = :slug limit 1;
```

The same query renders differently depending on `generate.validator`:

<Tabs>
<TabItem label="TypeScript only (default)">

```ts
import type {Client} from 'sqlfu';

const sql = `select id, slug, title, status from posts where slug = ? limit 1;`;
const query = (params: findPostBySlug.Params) => ({
  sql,
  args: [params.slug],
  name: 'findPostBySlug',
});

export const findPostBySlug = Object.assign(
  async function findPostBySlug(
    client: Client,
    params: findPostBySlug.Params,
  ): Promise<findPostBySlug.Result | null> {
    const rows = await client.all<findPostBySlug.Result>(query(params));
    return rows.length > 0 ? rows[0] : null;
  },
  {sql, query},
);

export namespace findPostBySlug {
  export type Params = {slug: string};
  export type Result = {
    id: number;
    slug: string;
    title: string | null;
    status: 'draft' | 'published';
  };
}
```

Plain types only: no validator import and no runtime checks.

</TabItem>
<TabItem label="arktype">

```ts
import {type Client, prettifyStandardSchemaError} from 'sqlfu';
import {type} from 'arktype';

const Params = type({
  slug: 'string',
});
const Result = type({
  id: 'number',
  slug: 'string',
  title: 'string | null',
  status: '"draft" | "published"',
});
const sql = `select id, slug, title, status from posts where slug = ? limit 1;`;
const query = (params: findPostBySlug.Params) => ({
  sql,
  args: [params.slug],
  name: 'findPostBySlug',
});

export const findPostBySlug = Object.assign(
  async function findPostBySlug(
    client: Client,
    params: findPostBySlug.Params,
  ): Promise<findPostBySlug.Result | null> {
    const parsedParamsResult = Params['~standard'].validate(params);
    if ('then' in parsedParamsResult) throw new Error('Unexpected async validation from Params.');
    if ('issues' in parsedParamsResult) throw new Error(prettifyStandardSchemaError(parsedParamsResult) || 'Validation failed');
    const rows = await client.all(query(parsedParamsResult.value));
    if (rows.length === 0) return null;
    const parsed = Result['~standard'].validate(rows[0]);
    if ('then' in parsed) throw new Error('Unexpected async validation from Result.');
    if ('issues' in parsed) throw new Error(prettifyStandardSchemaError(parsed) || 'Validation failed');
    return parsed.value;
  },
  {Params, Result, sql, query},
);

export namespace findPostBySlug {
  export type Params = typeof findPostBySlug.Params.infer;
  export type Result = typeof findPostBySlug.Result.infer;
}
```

Arktype schemas live in TS-syntax strings. Optional parameters appear as
`'title?': 'string'` on the key side.

</TabItem>

## Pretty errors

By default (`generate.prettyErrors: true`), validation failures throw an `Error`
whose message is a readable, indented issues list. The generated wrapper inlines
the Standard Schema result guard and calls `prettifyStandardSchemaError`
re-exported from `sqlfu`.

`prettifyStandardSchemaError` is vendored from
[trpc-cli](https://github.com/mmkal/trpc-cli/tree/main/src/standard-schema) and
re-exported as a reference implementation. The stable contract is the Standard
Schema `Result` shape, not sqlfu's prettifier.

### `prettyErrors: false`

Set `prettyErrors: false` to keep the raw Standard Schema issue list available
without running it through the prettifier:

```ts
generate: {
  validator: 'arktype',
  prettyErrors: false, // default: true
},
```

On failure, generated wrappers throw
`Object.assign(new Error('Validation failed'), {issues: result.issues})`. You
still get the issues array on `error.issues`.

In `prettyErrors: false` mode, the generated file has zero runtime dependency on
sqlfu. Only erased type imports are used. The wrapper is fully self-contained
apart from its validator library.

## One identifier per query

The function name is the identifier for everything related to the query:

```ts
import {findPostBySlug} from './sql/.generated/find-post-by-slug.sql.js';

const post = await findPostBySlug(client, {slug: 'hello'});

type P = findPostBySlug.Params;
type R = findPostBySlug.Result;

const paramsSchema = findPostBySlug.Params;
const resultSchema = findPostBySlug.Result;
const queryText = findPostBySlug.sql;
```

Namespace merging is what makes `findPostBySlug.Params` resolve as a value (the
schema) and a type. Consumers do not have to think about this: they write
`findPostBySlug.Params` in either position.

For queries with `update` semantics, the shape is `findPostBySlug.Data` +
`findPostBySlug.Params` + `findPostBySlug.Result`, matching the plain-TS output.

## Error behavior

Validation throws on invalid input. Callers who want recovery can call the
schemas directly via each library's safe-parse equivalent:

<Tabs>
<TabItem label="arktype">

```ts
import {type} from 'arktype';

const parsed = findPostBySlug.Params(userInput);
if (parsed instanceof type.errors) return handleError(parsed);
```

</TabItem>
<TabItem label="valibot">

```ts
import * as v from 'valibot';

const parsed = v.safeParse(findPostBySlug.Params, userInput);
if (!parsed.success) return handleError(parsed.issues);
```

</TabItem>
</Tabs>

The wrapper throwing by default is intentional. This is generated code, and the
right default is to fail loudly at the boundary.

## Not emitting a validator

If `generate.validator` is unset, `null`, or `undefined`, the generator emits the
plain TS output. There is no hybrid mode: a project picks one.

## Extending the generated shape

The generated file is readable and small. If you want a specific validator
refinement for a column, wrap the generated function in your application code:

```ts
import {type} from 'arktype';
import {findPostBySlug as rawFindPostBySlug} from './sql/.generated/find-post-by-slug.sql.js';

const RichParams = rawFindPostBySlug.Params.and(
  type({
    slug: /^[-a-z0-9]+$/,
  }),
);

export async function findPostBySlug(client: Client, params: typeof RichParams.infer) {
  const parsed = RichParams(params);
  if (parsed instanceof type.errors) throw parsed;
  return rawFindPostBySlug(client, parsed);
}
```

The generated schemas will never be richer than what a SQL type system can tell
us. Column-level refinement is an application concern today. Pluggable
validators and per-column overrides are
[planned](https://github.com/mmkal/sqlfu/blob/main/tasks/typegen-extensibility.md)
but not in scope yet.
