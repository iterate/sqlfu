# Better Auth adapter formats and appends when safe

## Status summary

Done. The Better Auth-managed SQL is formatted with sqlfu's formatter, first-time generation appends the managed section to compatible nonempty unfenced `definitions.sql` files after a scratch SQLite apply check, and unsafe first-time appends still fail clearly. Focused adapter/UI tests, formatting check, and sqlfu typecheck pass.

## Summary

The current `sqlfuBetterAuthAdapter` handles an empty `definitions.sql` and an existing `-- #region sqlfu:better-auth` section, but rejects a nonempty unfenced file. That makes the first Better Auth generation feel broken for normal sqlfu projects that already have application tables, for example:

```sql
create table someotherthing(id int, name text);
```

The adapter should instead make the first run "just work" when the existing application schema and the generated Better Auth schema are compatible. It should validate that by applying the combined desired schema to an empty scratch SQLite database. If that scratch load succeeds, append the Better Auth-managed section to the end of `definitions.sql`. If it fails, keep rejecting with a clear error so the user can add or fix the managed section manually.

## Assumptions

- "Fenced block" means the adapter's existing SQL comment region:

  ```sql
  -- #region sqlfu:better-auth
  -- generated by Better Auth through sqlfuBetterAuthAdapter; edit Better Auth config instead
  ...
  -- #endregion sqlfu:better-auth
  ```

- "Apply its changes cleanly" means the whole desired schema, `current definitions.sql` plus the generated Better Auth section, can be applied to a fresh scratch SQLite database. The adapter does not know the user's live database and should not attempt migration planning here.
- Existing application SQL outside the managed section should remain byte-for-byte unchanged. Formatting is only for the generated Better Auth SQL inside the managed section.
- Existing fenced files should continue to replace the managed section as before.
