**Status:** not started. This is a follow-up design task spun out of the typegen casing discussion. The current casing task should leave `sql/.generated/tables.ts` raw DB-shaped; this task asks whether generating schema-wide table/view row types is still a feature sqlfu wants.

# Reconsider generated table row types

## Why this exists

`sqlfu generate` currently emits `sql/.generated/tables.ts` with one row type per table/view. Those types are raw database shapes: `published_at`, not `publishedAt`. That makes sense if the file means "what SQLite hands back from `select *`", but it is different from the desired generated query boundary where query wrappers expose application-shaped camelCase data/results.

That split is coherent, but it raises a sharper product question: do schema-wide row types belong in sqlfu at all, or do they encourage app code to depend on broad raw table shapes instead of named query wrappers?

## Current decision for the casing task

Do not change `tables.ts` as part of generated query casing. The query wrapper is the explicit SQL-to-application boundary. `tables.ts`, if it remains, is raw DB-shaped.

## Questions to resolve

- [ ] Decide whether `tables.ts` should remain generated by default.
- [ ] Decide whether raw schema row types should move behind a config flag.
- [ ] Decide whether generated row types should be renamed to make their raw DB shape explicit.
- [ ] Decide whether docs should steer users away from importing broad table row types in application code.
- [ ] If the feature stays, document its intended use cases and non-use cases.
- [ ] If the feature goes, remove it without compatibility baggage.

## Notes

- Current implementation lives in `packages/sqlfu/src/typegen/index.ts` in `writeTablesFile`.
- The code comment currently says row types are "literally what SQLite hands back for `select * from <table>`."
- The library is pre-pre-alpha, so deleting this feature is acceptable if the design points that way.
